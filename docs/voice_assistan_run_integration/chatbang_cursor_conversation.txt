## Важливі ризики, які ще бачу

### 1. Audit queue може втрачати записи на завершенні процесу

`LiveAuditQueue` працює через daemon thread. У `main.py` у `finally` викликається `offer_engine_client.wait_for_shadow(...)`, але я не бачу явного `flush()` / `close()` audit queue для live mode. Це означає, що при швидкому виході частина audit JSONL може не встигнути записатися.

Це не критично для відповіді live, але критично для вашого головного продукту: post-session audit і claim ledger.

Я б додав:

```text id="86k389"
OfferEngineClient.close()
→ adapter.close()
→ audit_queue.flush()
→ audit_queue.close()
```

І викликав це у `main.py finally`.

### 2. Shadow mode state може бути не зовсім чесним

У shadow mode offer-engine генерує свою відповідь, але live користувач бачить стару LLM-відповідь. При цьому `OfferEngineLiveAdapter.generate_live_answer()` оновлює internal session state своїми shadow answers/claims. Якщо довго працювати в shadow, offer-engine session state буде описувати **не те, що реально було сказано live**, а те, що offer-engine “міг би сказати”.

Для shadow benchmark це може бути прийнятно, але треба явно логувати:

```text id="n88e9m"
spoken_answer_source: old_llm
shadow_answer_source: offer_engine
state_policy: shadow_state / spoken_state
```

І краще мати опцію:

```text id="w0wz8d"
--offer-shadow-state {isolated,shared}
```

За замовчуванням — `isolated`.

### 3. Live answer не стрімиться

У live mode зараз answer генерується повністю, потім друкується одним шматком. Це простіше й нормально для першої інтеграції. Але для відчуття realtime наступний етап — або швидке token-like printing, або latency measurement саме `transcript_final -> visible_answer_start`.

Зараз `first_token_at = offer_completed_at`, тобто latency summary не показує справжній “first visible token” для offer-engine. Це треба позначити окремою метрикою:

```text id="vh6gzk"
offer_engine_request_ms
offer_engine_visible_answer_ms
offer_engine_complete_ms
```

### 4. Quality gate може бути надто жорстким для коротких технічних питань

`live_quality_gate` reject-ить, якщо meaningful words < 5 і немає task/question intent. Але в реальному інтерв’ю бувають короткі follow-up:

```text id="4rmk8v"
“Яка модель?”
“Який baseline?”
“Скільки GPU?”
“Чому LightGBM?”
```

Такі питання можуть бути валідними, хоча короткі. Треба додати exception для коротких technical follow-ups, якщо є previous context.

### 5. Uploaded artifact містить `__pycache__`

Це packaging hygiene. Не страшно для функціональності, але суперечить “clean release”. Якщо agent package tool уже виключає pycache, треба завантажувати саме його output, а не робочу папку.
